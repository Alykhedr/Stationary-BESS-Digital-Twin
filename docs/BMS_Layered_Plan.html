0. Design Philosophy — The Two Rules That Drive Everything

Rule 1 — Truth vs Belief. The BMS is an observer: it only sees measurements, never the plant's truth. It reads V, I, T through sensor models and estimates SOC/SOH. The error SOC_true − SOC_est is itself a primary digital-twin output.

Rule 2 — Layers, not shared functions. Each layer talks only to its neighbours through one defined bus. Each layer has exactly one owner. The single coupling point between the two of us is the Layer 2→3 handshake: SOC_est, Q_est, R_est. We agree on interfaces first, build in parallel, and integrate only at planned milestones.

Three hard architecture rules (carried over from v1):

  1. The BMS never reads SOC_true from the plant. Truth signals are logged for error analysis only — firewalled from all BMS inputs.
  2. The EMS never talks to the battery directly. Every current request passes through the BMS limit chain. The static clamps in current_limiter.m (SOC window, fixed I limits) move into the BMS and become dynamic.
  3. The plant always obeys physics, never the BMS's beliefs. If the BMS believes SOC = 20% but truth is 5%, the voltage collapses and the BMS must catch it via the voltage protection path — exactly as in reality.

5. Layer 3–4 Function Specifications + Integration ALI

5.1 bms_limits.m — dynamic State-of-Power (stateless, pure math)

function [I_ch_max, I_dis_max] = bms_limits(SOC_est, Q_est, R_est_ch, R_est_dis, ...
                                            V_meas, T_meas, dt_h, cfg)
% Each limit = min over independent constraint curves (all ≥ 0):
%
% CHARGE:
%   I_ch_volt = (V_max_protect − V_meas_max) / R_est_ch        % voltage headroom
%   I_ch_temp = f_derate_T_chg(T_meas) · I_chg_max_cont        % piecewise-linear ramps
%   I_ch_soc  = max(0, window_max − SOC_est) · Q_est / Δt      % window (replaces old clamp)
%   I_ch_max  = max(0, min([I_ch_volt, I_ch_temp, I_ch_soc, I_chg_max_cont]))
%
% DISCHARGE (symmetric):
%   I_dis_volt = (V_meas_min − V_min_warn) / R_est_dis
%   I_dis_temp = f_derate_T_dis(T_meas) · I_dis_max_cont
%   I_dis_soc  = max(0, SOC_est − window_min) · Q_est / Δt
%
% Temperature derate (charge example): 0 below 0 °C → ramp to 1 over [0,5] →
%   1 in [5,40] → ramp to 0 over [40,45]. Worst cell governs at pack phase
%   (T_meas_max for hot end, T_meas_min for cold end).
%
% NOTE: protection inputs V_meas/T_meas are RAW measurements, not estimates —
%       deliberate: the safety path must not depend on estimator health.
% The voltage-headroom term implicitly creates CC→CV-like taper near full —
% no explicit CV controller needed at this timescale.

6. The Timescale Problem — and the Dual-Rate Solution

The mismatch: a real BMS samples at 1–100 Hz and its protection path reacts in milliseconds. Our simulation steps at Δt = 1 h (needed to make a 10-year aging run tractable: 87,600 steps). Naively this means protection acts one full hour after a threshold crossing, and rest-detection / lag dynamics are quantized to whole hours.

6.1 The key design decision that solves it

Every BMS function takes dt_h as an input parameter — no function hard-codes the timestep. The estimators, limits, and mode machine are rate-agnostic MATLAB functions: the SAME code runs at Δt = 1 h inside the 10-year Simulink model AND at Δt = 0.01 s (100 Hz) inside a script-driven test harness. Two execution contexts, one codebase, zero duplicated logic.

6.2 The two execution contexts

Context A — Longevity twin (Simulink)Context B — High-rate harness (MATLAB script)
Timestep1 h10 ms – 1 s (up to 100 Hz)
Horizon10 yearsminutes – a few days
PurposeEnergy, aging, SOC/SOH estimation drift, dispatch interactionProtection reaction timing, mode-transition dynamics, hysteresis/chattering, sensor lag (τ_T = 30 s becomes resolvable), rest-detection timing
Plant signal sourceFull plant model in the loopSynthetic high-rate profiles (current steps, voltage ramps, fault injections) — plant physics not needed for protection-path validation
What it cannot doResolve sub-hour protection eventsAging (timescale too short — irrelevant at minutes)

6.3 What this buys us

  • The thesis claim upgrades from "protection logic exists but is untested at realistic timescales" to "the identical protection code is validated at 100 Hz in a dedicated harness, then deployed at 1 h in the longevity twin."
  • The self-test scripts (test_mode.m, test_limits.m) simply run the harness at both rates — chattering and hysteresis-band bugs that are invisible at 1 h show up immediately at 100 Hz.
  • Sensor lag (τ_T = 30 s) and rest-detection (2 h) are expressed in hours in cfg and scale automatically with dt_h — no unit surgery between contexts.
  • Remaining documented limitation: the coupled plant+BMS simulation still cannot resolve sub-hour events (short circuit, inrush) — Context B validates the BMS side alone. Coupling a high-rate plant model is a possible later phase (a 1 s plant variant over representative days), listed in the roadmap §10.

8. The Pack Phase — Building & Testing (Sprint 2 → M2)

The step after the single-cell BMS is validated (M1) is scaling the plant to pack level, following Schimpe's Energy Neighbor hierarchy. Approach: vectorize the existing validated cell model — NOT Simscape Battery Pack Builder, which would replace our Schimpe physics with a generic equivalent circuit and discard the 57-check validation.

8.1 Scaling hierarchy (Schimpe Table 2.2/2.3)

cell (3.0 Ah, 3.2 V)
  └─ ×12 parallel  → block   (1s12p, 36 Ah)   ← lumped: one R, one OCV, one thermal node
       └─ ×16 series  → module  (16s12p, ~51 V)  ← the BMS slave's domain: 16 V-taps, 2–4 NTC
            └─ ×13 series → string/rack (~665 V)  ← the BMS master's domain: 1 current sensor

The simulation models ONE module (16 blocks) explicitly; rack level is algebraic scaling (×13 series voltage, current identical in a series string). Lumping 12 parallel cells into one block is Schimpe's own simplification — parallel cells share a voltage tap, intra-block imbalance is not observable by the real BMS either.

8.2 Build steps (plant side)

  1. Vectorize state: SOC, V, R, T, aging states go from scalar → [16×1]. The physics functions already operate element-wise — the change is dimension declaration + broadcasting, not new equations.
  2. Series electrical coupling: same string current through all blocks; module voltage = Σ V_block. The weakest block's voltage hits limits first — this is exactly what the worst-cell protection logic must catch.
  3. Per-block parameter scatter (frozen at init, seeded → reproducible): Q_block ~ N(36 Ah, σ=0.5 %), R_block ~ N(nom, σ=2 %), self-discharge ~ N(nom, σ=5 %). Self-discharge scatter is the dominant long-run divergence driver — without scatter, balancing would silently never trigger (wrong answer for a 10-year efficiency study).
  4. Thermal: per-block 0D nodes with a common ambient; optionally a simple block-to-block coupling. Temperature sensors are SPARSE (2–4 NTC per module, not 16) — sensor-placement sparsity is realistic and matters for worst-cell detection.
  5. Aging per block: calendar + cycle aging run per block on its own SOC/T trajectory → blocks age apart over years, feeding back into divergence.

8.3 Build steps (BMS side)

  1. Slave layer: sensor models become vector instances (16 V-taps, 2–4 T). bms_balancing.m goes live: balance block k if V_meas[k] > min(V_meas) + 5 mV AND mode ∈ {CHARGE, STANDBY}; I_bleed = V/33 Ω ≈ 100 mA; bleed heat to slave PCB (not the cell thermal node); E_bal accumulator.
  2. Master aggregation: worst-cell selection — V_meas_max, V_meas_min, T_meas_max, T_meas_min across blocks feed the SAME §5.1 limit equations. No equation changes, only input aggregation.
  3. Estimation at pack: string SOC from string current + worst/mean block voltages; per-block SOC estimation optional (thesis decision — real systems differ here).
  4. Balance feedback into plant: block SOC derivative gains −I_bleed[k]/Q_block.
